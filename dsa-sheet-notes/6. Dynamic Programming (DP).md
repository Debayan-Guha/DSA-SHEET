# Fibonacci Recursion

## 1. Recursive Fibonacci

```java
int fib(int n) {
    if(n <= 1) {
        return n;   // fib(0)=0, fib(1)=1
    }
    return fib(n-1) + fib(n-2);
}
```

### Base Cases (Both are Equivalent)

```java
// Version 1 - Cleaner
if(n <= 1) return n;
```

```java
// Version 2 - Explicit
if(n == 0 || n == 1) return n;
```

**Note:** Fibonacci is defined only for `n ≥ 0`, so we never pass negative values.

---

## 2. Visual Tree for fib(5)

```text
                                fib(5)
                       __________/ \__________
                      /                         \
                  fib(4)                     fib(3)
                ____/ \____               ____/ \____
               /           \             /           \
           fib(3)       fib(2)       fib(2)       fib(1)
          ___/ \___     /     \      /     \
         /         \   /       \    /       \
     fib(2)     fib(1) fib(1) fib(0) fib(1) fib(0)
      /   \
     /     \
 fib(1)   fib(0)
```

---

## 3. The Overlapping Subproblems Problem ⚠️

Same subproblems are computed multiple times.

From the tree above:

- `fib(3)` is computed **2 times**
- `fib(2)` is computed **3 times**
- `fib(1)` is computed **5 times**
- `fib(0)` is computed **3 times**

### Why is this inefficient?

- **Time Complexity:** `O(2ⁿ)` (Exponential)
- **Space Complexity:** `O(n)` (Recursion stack)

For `n = 50`, recursion becomes extremely slow.

### Example of Redundancy

```text
fib(5) calls fib(4) and fib(3)

fib(4) calls fib(3) and fib(2)    ← fib(3) computed again

fib(3) calls fib(2) and fib(1)    ← fib(2) computed again
```

---


## 4. Solution: Memoization (Top-Down DP)


### A. Idea

Instead of solving every subproblem repeatedly, we store the answer of each subproblem the first time it is computed and reuse it whenever needed again.

```text
fib(5)
↓
fib(4)
↓
fib(3)
↓
fib(2)
↓
fib(1), fib(0)
```

When the recursion encounters the same subproblem again, it does not recompute it.

```text
First call to fib(3)
        ↓
Compute and store result

Second call to fib(3)
        ↓
Use stored result (Cache Hit)
```

Thus, each state is computed only once.

Memoization follows a **Top-Down Dynamic Programming** approach because we start from the original problem `fib(n)` and recursively move toward the base cases.

```text
Direction:

5 → 4 → 3 → 2 → 1 → 0
```

---

### B. Steps

**Step 1: Identify the Need for Caching**

We are computing the same values repeatedly.

- Store results when first computed.
- Reuse stored values later.

---

**Step 2: Create a Cache**

```java
int[] memo = new int[n+1];
Arrays.fill(memo, -1);
```

- Size = `n+1` because index = value of `n`.
- `-1` means **"not computed yet"**.

**Why size = n+1 ?**

To store `fib(5)` at index `5`.

Array indices:

```text
0 1 2 3 4 5
```

Index `5` must exist.

---

**Step 3: Modify the Recursive Function**

```java
int fib(int n) {

    // Base case
    if(n <= 1)
        return n;

    // Cache hit
    if(memo[n] != -1)
        return memo[n];

    // Compute and store
    memo[n] = fib(n-1) + fib(n-2);

    return memo[n];
}
```

---

### C. Complete Memoization Code

```java
class Solution {

    public int fib(int n) {

        int[] memo = new int[n+1];
        Arrays.fill(memo, -1);

        return dp(n, memo);
    }

    private int dp(int n, int[] memo) {

        if(n <= 1)
            return n;

        if(memo[n] != -1)
            return memo[n];

        return memo[n] = dp(n-1, memo) + dp(n-2, memo);
    }
}
```

---

### D. Step-by-Step Dry Run for fib(5)

**Initial State**

```text
memo = [-1, -1, -1, -1, -1, -1]
```

**Execution Trace**

| Step | Call | Action | memo after |
|------|------|------|------|
|1|dp(5)|memo[5]==-1 → compute|[-1,-1,-1,-1,-1,-1]|
|2|dp(4)|memo[4]==-1 → compute|[-1,-1,-1,-1,-1,-1]|
|3|dp(3)|memo[3]==-1 → compute|[-1,-1,-1,-1,-1,-1]|
|4|dp(2)|memo[2]==-1 → compute|[-1,-1,-1,-1,-1,-1]|
|5|dp(1)|Base case → return 1|[-1,-1,-1,-1,-1,-1]|
|6|dp(0)|Base case → return 0|[-1,-1,-1,-1,-1,-1]|

|8|Back to dp(3)|memo[3]=dp(2)+dp(1)| |

|10|dp(1)|Base case → return 1|[-1,-1,1,-1,-1,-1]|
|11|Back to dp(3)|memo[3]=1+1=2|[-1,-1,1,2,-1,-1]|
|12|Back to dp(4)|memo[4]=dp(3)+dp(2)| |

|14|dp(2)|CACHE HIT → return 1|[-1,-1,1,2,-1,-1]|

|16|Back to dp(5)|memo[5]=dp(4)+dp(3)| |

|18|dp(3)|CACHE HIT → return 2|[-1,-1,1,2,3,-1]|
|19|Back to dp(5)|memo[5]=3+2=5|[-1,-1,1,2,3,5]|

**Final Result**

```text
fib(5) = 5
```

✅ Answer = **5**

---

### E. Key Points to Remember

- Memoization is a **Top-Down DP** approach.
- Uses recursion.
- Array size = `n+1` because we need index `n`.
- `memo[0]` and `memo[1]` remain `-1`.
- Cache checking happens before computation.
- Every state is computed only once.
- Time complexity improves from `O(2ⁿ)` to `O(n)`.
- Space complexity remains `O(n)`.

---

### F. Complexity Comparison

| Approach | Time Complexity | Space Complexity |
|------------|---------------|----------------|
| Plain Recursion | O(2ⁿ) | O(n) |
| Memoization | O(n) | O(n) |

**Example: n = 50**

**Plain Recursion**

```text
≈ 2^50 operations
(impractical)
```

**Memoization**

```text
≈ 50 operations
(instant)
```

### G. Alternative: Using HashMap

**Code**

```java
class Solution {

    Map<Integer, Integer> memo = new HashMap<>();

    public int fib(int n) {

        if(n <= 1)
            return n;

        if(memo.containsKey(n))
            return memo.get(n);

        memo.put(n, fib(n-1) + fib(n-2));

        return memo.get(n);
    }
}
```

### I. Important Observation

Unlike the array solution:

```java
int[] memo = new int[n+1];
```

which creates

```text
[-1, -1, 1, 2, 3, 5]
```

the `HashMap` stores **only the values that are actually computed**:

```text
{2=1, 3=2, 4=3, 5=5}
```

Notice that `0` and `1` are **not stored** because they are base cases and are returned directly.

### J. When to Use Array vs HashMap?

| Array | HashMap |
|---------|---------|
| Small and contiguous indices | Sparse or irregular indices |
| Faster access | More flexible |
| Less memory overhead | Useful for large values |
| Best for Fibonacci | Useful in many DP problems |

## 5. Solution: Tabulation (Bottom-Up DP)

### A. Idea

Instead of solving recursively and storing answers later, we start from the base cases and build the answer iteratively.

```text
fib(0) = 0
fib(1) = 1

fib(2) = fib(1) + fib(0)
fib(3) = fib(2) + fib(1)
fib(4) = fib(3) + fib(2)
...
fib(n) = fib(n-1) + fib(n-2)
```

Unlike Memoization (Top-Down), Tabulation works from **Bottom-Up**.

---

### B. Steps

**Step 1: Create a DP Array**

```java
int[] dp = new int[n+1];
```

Size = `n+1` because we need indices from `0` to `n`.

---

**Step 2: Initialize Base Cases**

```java
dp[0] = 0;
dp[1] = 1;
```

These values are already known.

---

**Step 3: Fill the Table**

```java
for(int i = 2; i <= n; i++) {
    dp[i] = dp[i-1] + dp[i-2];
}
```

Each value depends on the previous two values.

---

**Step 4: Return the Answer**

```java
return dp[n];
```

---

### C. Complete Tabulation Code

```java
class Solution {

    public int fib(int n) {

        if(n <= 1)
            return n;

        int[] dp = new int[n+1];

        dp[0] = 0;
        dp[1] = 1;

        for(int i = 2; i <= n; i++) {
            dp[i] = dp[i-1] + dp[i-2];
        }

        return dp[n];
    }
}
```

---

### D. Step-by-Step Dry Run for fib(5)

**Initial State**

```text
dp = [0, 1, 0, 0, 0, 0]
```

---

**Execution Trace**

| Iteration | i | Formula | Value Stored | dp Array |
|------------|---|----------|--------------|----------|
|1|2|dp[1]+dp[0]|1+0=1|[0,1,1,0,0,0]|
|2|3|dp[2]+dp[1]|1+1=2|[0,1,1,2,0,0]|
|3|4|dp[3]+dp[2]|2+1=3|[0,1,1,2,3,0]|
|4|5|dp[4]+dp[3]|3+2=5|[0,1,1,2,3,5]|

---

**Growth of the DP Array**

```text
Initially

[0, 1, 0, 0, 0, 0]

After i = 2

[0, 1, 1, 0, 0, 0]

After i = 3

[0, 1, 1, 2, 0, 0]

After i = 4

[0, 1, 1, 2, 3, 0]

After i = 5

[0, 1, 1, 2, 3, 5]
```

---

**Final Result**

```text
fib(5) = dp[5]
        = 5
```

✅ Answer = **5**

---

### E. Visualization of Table Filling

```text
Index : 0   1   2   3   4   5
--------------------------------
Value : 0   1   1   2   3   5
            ↑
      Base Cases

2 = 1 + 0
3 = 1 + 1
4 = 2 + 1
5 = 3 + 2
```

---

### F. Key Points to Remember

- Tabulation is a **Bottom-Up DP** approach.
- No recursion is used.
- No recursion stack is required.
- Start with base cases and build towards `n`.
- Every state is computed exactly once.
- The answer is stored in `dp[n]`.

---

### G. Complexity Comparison

| Approach | Time Complexity | Space Complexity |
|------------|---------------|----------------|
| Recursion | O(2ⁿ) | O(n) |
| Memoization | O(n) | O(n) |
| Tabulation | O(n) | O(n) |

---

### H. Memoization vs Tabulation

| Memoization (Top-Down) | Tabulation (Bottom-Up) |
|------------------------|------------------------|
| Uses recursion | Uses loops |
| Starts from n and goes downward | Starts from base cases |
| Requires recursion stack | No recursion stack |
| Computes only required states | Usually computes all states |
| Easier to write | Usually faster |
| Top-Down approach | Bottom-Up approach |

---

### I. Observation

Memoization:

```text

 ↓
fib(4)
 ↓
fib(3)
 ↓
fib(2)
 ↓
fib(1), fib(0)
```

Direction:

```text
5 → 4 → 3 → 2 → 1 → 0
```

Tabulation:

```text
fib(0), fib(1)
      ↓
fib(2)
      ↓
fib(3)
      ↓
fib(4)
      ↓
fib(5)
```

Direction:

```text
0 → 1 → 2 → 3 → 4 → 5
```

### J. Space Optimized Version

#### Idea

In normal Tabulation, we store all Fibonacci values:

```text
[0, 1, 1, 2, 3, 5]
```

However,

```text
fib(i) = fib(i-1) + fib(i-2)
```

depends only on the previous two values.

Therefore, instead of storing the entire array, we can store only:

```text
prev2 = fib(i-2)
prev1 = fib(i-1)
```

and compute

```text
curr = prev1 + prev2
```

Thus, space complexity improves from:

```text
O(n)
```

to

```text
O(1)
```

---

#### Steps

**Step 1: Handle Base Cases**

```java
if(n <= 1)
    return n;
```

**Step 2: Store the First Two Fibonacci Numbers**

```java
int prev2 = 0;
int prev1 = 1;
```

**Step 3: Compute Remaining Values**

```java
for(int i = 2; i <= n; i++) {

    int curr = prev1 + prev2;

    prev2 = prev1;
    prev1 = curr;
}
```

**Step 4: Return the Answer**

```java
return prev1;
```

---

#### Complete Space Optimized Code

```java
class Solution {

    public int fib(int n) {

        if(n <= 1)
            return n;

        int prev2 = 0;
        int prev1 = 1;

        for(int i = 2; i <= n; i++) {

            int curr = prev1 + prev2;

            prev2 = prev1;
            prev1 = curr;
        }

        return prev1;
    }
}
```

---

#### Step-by-Step Dry Run for fib(5)

**Initial State**

```text
prev2 = 0
prev1 = 1
```

**Execution Trace**

| Iteration | i | curr = prev1 + prev2 | prev2 after | prev1 after |
|------------|---|---------------------|-------------|-------------|
|1|2|1+0=1|1|1|
|2|3|1+1=2|1|2|
|3|4|2+1=3|2|3|
|4|5|3+2=5|3|5|

**Variable Movement**

```text
Initially

prev2 = 0
prev1 = 1

----------------

i = 2

curr = 1

prev2 = 1
prev1 = 1

----------------

i = 3

curr = 2

prev2 = 1
prev1 = 2

----------------

i = 4

curr = 3

prev2 = 2
prev1 = 3

----------------

i = 5

curr = 5

prev2 = 3
prev1 = 5
```

**Final Result**

```text
fib(5) = 5
```

✅ Answer = **5**

---

#### Visualization

```text
Initial

prev2 = 0
prev1 = 1

           curr
            ↓
0    1    (1)

prev2 prev1


After i = 3

1    1    (2)

prev2 prev1


After i = 4

1    2    (3)

prev2 prev1


After i = 5

2    3    (5)

prev2 prev1
```

---

#### Key Points to Remember

- Still a Bottom-Up DP approach.
- No recursion is used.
- No DP array is required.
- Only two variables are maintained.
- Every state is computed exactly once.
- Space complexity becomes `O(1)`.

---

#### Complexity Comparison

| Approach | Time Complexity | Space Complexity |
|------------|---------------|----------------|
| Recursion | O(2ⁿ) | O(n) |
| Memoization | O(n) | O(n) |
| Tabulation | O(n) | O(n) |
| Space Optimized Tabulation | O(n) | O(1) |

---

#### Observation

Normal Tabulation stores:

```text
[0,1,1,2,3,5]
```

Space Optimized Tabulation stores only:

```text
prev2 = 3
prev1 = 5
```

Thus, memory usage becomes constant.


## Relationship Between DP and Space Optimization

Dynamic Programming mainly has two approaches:

```text
1. Memoization (Top-Down DP)
2. Tabulation (Bottom-Up DP)
```

```text
Recursion
     ↓
Memoization (Top-Down DP)
     ↓
Tabulation (Bottom-Up DP)
     ↓
Space Optimization
```

---

### Memoization (Top-Down DP)

- Uses recursion.
- Stores already computed answers.
- Avoids repeated calculations.
- Computes only required states.

```text
fib(5)
 ↓
fib(4)
 ↓
fib(3)
 ↓
fib(2)
 ↓
fib(1), fib(0)
```

---

### Tabulation (Bottom-Up DP)

- Uses loops.
- Starts from the base cases.
- Builds the solution iteratively.
- Stores all states in a DP table.

```text
fib(0), fib(1)
      ↓
fib(2)
      ↓
fib(3)
      ↓
fib(4)
      ↓
fib(5)
```

---

### Is Space Optimization a DP Approach?

**No.**

Space Optimization is **not a separate Dynamic Programming approach**.

Dynamic Programming has only two approaches:

```text
1. Top-Down DP (Memoization)
2. Bottom-Up DP (Tabulation)
```

Space Optimization is simply an **optimization applied to a Bottom-Up DP solution**.

---

### Why Space Optimization is Possible

In Tabulation, we usually store all states.

For Fibonacci:

```text
dp[0] = 0
dp[1] = 1
dp[2] = 1
dp[3] = 2
dp[4] = 3
dp[5] = 5
```

Entire DP array:

```text
[0, 1, 1, 2, 3, 5]
```

But notice:

```text
dp[i] = dp[i-1] + dp[i-2]
```

To calculate `dp[i]`, we only need:

- `dp[i-1]`
- `dp[i-2]`

Therefore, storing the whole array is unnecessary.

Instead of:

```java
int[] dp = new int[n+1];
```

we can use only two variables:

```java
int prev2 = 0;
int prev1 = 1;
```

and compute:

```java
int curr = prev1 + prev2;
```

Thus, the DP array is eliminated.

---

### Space Complexity Improvement

| Solution | Time Complexity | Space Complexity |
|------------|---------------|----------------|
| Recursion | O(2ⁿ) | O(n) |
| Memoization | O(n) | O(n) |
| Tabulation | O(n) | O(n) |
| Space Optimized Tabulation | O(n) | O(1) |

---

### Important Observation

Space Optimization does **not** create a new algorithm.

It simply reduces memory usage of an existing Bottom-Up DP solution.

Therefore,

```text
Space Optimization
        ⊂
Bottom-Up DP (Tabulation)
        ⊂
Dynamic Programming
```

or equivalently,

```text
Dynamic Programming
│
├── Memoization (Top-Down DP)
│
└── Tabulation (Bottom-Up DP)
      │
      └── Space Optimization
```

### Conclusion

- **Memoization** and **Tabulation** are the two fundamental Dynamic Programming approaches.
- **Space Optimization is not a third DP approach.**
- It is an optimization technique applied to a Tabulation solution whenever only a few previous states are required.
- Therefore, Space Optimization can be viewed as a **special case of Bottom-Up DP**.

## How to Identify Whether a Problem Can Be Solved Using DP?

Dynamic Programming is usually applied when:

**1. We Need to Try All Possible Ways**

Examples:

- Pick or not pick an element.
- Move left or right.
- Partition an array in different ways.
- Choose one among multiple options.

```text
Try all possible choices
        ↓
Find answers for each choice
        ↓
Combine them
```

---

**2. We Need an Optimal Answer**

Questions containing:

- Minimum
- Maximum
- Best
- Longest
- Shortest
- Largest
- Smallest
- Minimum cost
- Maximum profit

Examples:

```text
Minimum Path Sum
Maximum Profit
Longest Common Subsequence
Shortest Path
```

---

**3. The Problem Asks for Counting**

Questions containing:

- Number of ways
- Count ways
- Total ways
- Distinct ways

Examples:

```text
Count number of subsets
Climbing Stairs
Coin Change II
Unique Paths
```

---

**4. Recursive Solutions Produce Overlapping Subproblems**

Same subproblem is solved repeatedly.

```text
fib(5)
 ├── fib(4)
 │     └── fib(3)
 └── fib(3)
```

Here, `fib(3)` is computed multiple times.

This indicates that DP can be applied.

---

**5. Problem Can Be Broken into Smaller Subproblems**

```text
Original Problem
        ↓
Smaller Problems
        ↓
Even Smaller Problems
        ↓
Base Case
```

Examples:

```text
fib(5)
depends on

fib(4) and fib(3)

fib(4)
depends on

fib(3) and fib(2)
```

---

## Two Main Properties of DP

### 1. Optimal Substructure

The optimal answer of the larger problem can be constructed from optimal answers of smaller subproblems.

```text
Answer(n)
depends on

Answer(smaller problems)
```

---

### 2. Overlapping Subproblems

Same subproblem appears multiple times.

```text
fib(3)
fib(2)
fib(3)
fib(2)
```

Instead of recomputing them, store the answers.

---

## General Pattern

```text
Can I make a choice?
        ↓
Can I try all possible choices?
        ↓
Do I need the best answer or count all answers?
        ↓
Will recursion solve the same subproblem multiple times?
        ↓
YES
        ↓
Dynamic Programming
```

---

## Important Note

Not every recursive problem is a DP problem.

```text
Recursion
│
├── Simple recursion
│     (Factorial, Binary Search)
│
└── Recursion with overlapping subproblems
      ↓
      Dynamic Programming
```

### Rule of Thumb

```text
Try all possible ways
        +
Find best answer / count ways
        +
Overlapping subproblems
        =
Dynamic Programming
```
